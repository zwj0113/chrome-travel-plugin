Chrome 插件架构方案

选择最适合旅游攻略信息收集场景的架构

方案 A：内容脚本集中式
YouTube 内容脚本
DOM解析 + 视频下载 + 评论抓取 + API调用 + Markdown生成
B站 内容脚本
DOM解析 + 视频下载 + 评论抓取 + API调用 + Markdown生成
小红书 内容脚本
DOM解析 + 图片识别 + 评论抓取 + API调用 + Markdown生成
Background
仅做 API 转发

A. 内容脚本集中式

每个平台一个"大而全"的内容脚本，承担所有逻辑

方案 B：适配器 + 流水线架构 ⭐ 推荐
YouTube 适配器
仅DOM提取
B站 适配器
仅DOM提取
小红书 适配器
仅DOM提取
⬇
Background 流水线编排
提取 → 下载 → 转录 → 总结 → 生成MD
AI 服务层
硅基流动/DeepSeek
降级服务
Whisper 本地

B. 适配器 + 流水线 ⭐

平台适配器只做 DOM 提取，Background 编排处理流水线

方案 C：Offscreen + Worker 架构
内容脚本 — DOM提取
⬇
Service Worker — 编排调度
⬇
Offscreen Doc
音频处理
Web Worker
Whisper推理

C. Offscreen + Worker

利用 Chrome 新 API 做重型处理，更现代但更复杂

对比总结

A. 集中式 B. 适配器+流水线 C. Offscreen
可扩展性 低 — 加平台需改大脚本 高 — 加个适配器即可 高
维护难度 高 — 逻辑耦合 低 — 职责分离 中 — 新API不成熟
本地处理能力 中 中 — 需Offscreen辅助 强 — 原生支持
开发复杂度 低 中 高
Whisper降级支持 难实现 可扩展接入 原生支持

💡 推荐：方案 B

理由：平台适配器只负责 DOM 提取（薄层），核心逻辑（AI调用、流水线、Markdown生成）集中在 Background，职责清晰。新增平台只需加一个适配器。Whisper 本地降级可以通过 Offscreen Document 按需扩展，不影响主架构。
